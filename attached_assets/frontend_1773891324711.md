# Sage — Frontend Developer Reference

> **Who this is for:** The frontend developer building the React SPA for Sage.  
> **What Sage is:** An offline-first, desktop AI study assistant for university students. It runs entirely on the user's Windows machine — no internet required for core features. The backend is a Python process (FastAPI + LangGraph + local LLM), the frontend is a React SPA served from that same process.

---

## 1. Project Context

**Institution:** Thal University Bhakkar — Department of Computer Science & IT  
**Programs:** BSSE · BSCS · BSIT (3 programs, 8 semesters each)  
**End users:** University students using their personal/lab Windows PCs

---

## 2. Runtime Architecture

```text
Windows Machine
│
├── Process A  →  Python (uvicorn + FastAPI)                 :8765
│     ├── FastAPI: serves React static files + all API routes
│     ├── LangGraph: 8-intent agent graph
│     ├── ChromaDB: vector store (two collections)
│     │     ├── curriculum        (pre-indexed course materials)
│     │     └── user_uploads      (student-uploaded documents)
│     └── SQLite (sage.db): conversation checkpoints + long-term memory
│
└── Process B  →  llama-server (C++ binary, Qwen3.5-4B GGUF)  :random port
      Started as subprocess by Process A on startup.
      Process A talks to it via HTTP: ChatOpenAI → localhost:{port}/v1
```

**The frontend (React) only talks to Process A on port 8501.**  
It never talks to Process B directly. All LLM interaction is encapsulated inside the Python layer.

**Port:** `8765` (default, configurable in `config/default.toml` under `[ui]`)  
**Base URL in production:** `http://localhost:8765`

## 3. Agent Modes (Intents)

These are the exact 8 modes the backend understands. The UI must expose all of them.

| Mode ID (send in API) | Display Name | Emoji | Description |
| --- | --- | --- | --- |
| `general` | General | 🤖 | Answers from model's parametric knowledge + session history. **No RAG retrieval.** |
| `explain` | Explain | 💬 | Concept explanation with RAG retrieval + citations |
| `quiz` | Quiz Me | 🧩 | Generates MCQ/short-answer questions from course material |
| `diagram` | Diagram | 📊 | Generates a Mermaid diagram (flowchart / sequence / etc.) |
| `roadmap` | Study Plan | 📅 | Builds a structured study schedule |
| `research` | Research | 🔬 | Deep-dive with optional online search (arXiv, Wikipedia) |
| `fix` | Fix Code | 🔧 | Code debugging, fix suggestions, sandbox execution |
| `thinking` | Thinking | 🧠 | Chain-of-thought extended reasoning (maps to `general` with thinking_mode=true) |

## 4. Course Filter

Courses are not a fixed hardcoded list. They are discovered from the filesystem at ingest time and stored as metadata in ChromaDB. The `/api/courses` endpoint (see §6) returns the available list dynamically.

**However,** the UI must always show an **"All Courses"** option (value: `"all"`) in addition to the returned list. When `"all"` is selected, retrieval runs across the entire curriculum collection with no course-code filter.

**Example values you will receive from the API:**

```json
["CS101", "CS201", "CS202", "CS301", "CS302", "CS303", "CS401", "CS402"]
```

The exact list depends on what has been ingested. Build the course selector to be dynamic, not hardcoded.

## 5. Conversation Persistence Model

Each conversation has a **thread_id** (UUID4, 8-char prefix used for display).

| Field | Type | Description |
| --- | --- | --- |
| `thread_id` | `string` | Unique conversation identifier |
| `title` | `string` | Auto-generated after first exchange (5-word summary) |
| `last_message_preview` | `string` | First 100 chars of last assistant response |
| `updated_at` | `string` (ISO 8601) | Timestamp of last activity |

**Grouping for sidebar display:**

- **Today** — `updated_at` is today
- **Yesterday** — `updated_at` is yesterday  
- **This Week** — 2–7 days ago
- **Last Month** — 8–30 days ago
- (anything older is not shown by default)

**Max conversations stored:** 100 (oldest auto-pruned by backend)

## 6. API Reference

**Base URL:** `http://localhost:8765`  
**All endpoints:** prefix `/api/`  
**Content-Type:** `application/json` unless stated otherwise

### 6.1 `POST /api/chat` — Submit a message

Submits a user message. Returns a `thread_id` and `message_id`. The actual response content is streamed separately via `GET /api/stream/{thread_id}`.

**Call this first, then immediately open the SSE stream.**

```text
POST /api/chat
Content-Type: application/json
```

**Request body:**

```json
{
  "thread_id": "abc12345",
  "message": "Explain binary search trees with examples",
  "mode": "explain",
  "course": "CS201"
}
```

| Field | Type | Required | Notes |
| --- | --- | --- | --- |
| `thread_id` | `string \| null` | No | `null` to start a new conversation |
| `message` | `string` | Yes | Max 2000 chars (enforced by backend) |
| `mode` | `string` | Yes | One of the 8 mode IDs from §3 |
| `course` | `string` | Yes | Course code or `"all"` |

**Response `200 OK`:**

```json
{
  "thread_id": "abc12345",
  "message_id": "msg_7f3a"
}
```

| Field | Notes |
| --- | --- |
| `thread_id` | If input was `null`, a new thread_id is created and returned here — store it |
| `message_id` | Used to correlate the stream to this specific exchange |

**Response `503 Service Unavailable`:**

```json
{ "detail": "Model not ready. Please wait." }
```

Show a "Model is loading, please wait" state in the UI. Poll `/api/status` to know when to retry.

---

### 6.2 `GET /api/stream/{thread_id}` — Stream the response (SSE)

Opens a Server-Sent Events stream. The response is delivered as a sequence of events.

```text
GET /api/stream/abc12345
Accept: text/event-stream
```

**React usage:**

```typescript
const source = new EventSource(`/api/stream/${threadId}`);
source.onmessage = (e) => {
  const data = JSON.parse(e.data);
  // handle based on data.type
};
```

**Event types:**

| `data.type` | Other fields | Meaning |
| --- | --- | --- |
| `"chunk"` | `"text": "..."` | Append this text to the last assistant message bubble |
| `"done"` | — | Stream is complete. Close EventSource. |
| `"error"` | `"message": "..."` | An agent error occurred. Display the message. Close EventSource. |
| `"tool_call"` | `"name": "..."` | (Optional to display) Agent is calling a tool, e.g. `"corpus_search"`, `"calculator"` |
| `"thinking"` | `"text": "..."` | (Optional to display) Internal reasoning step — show as greyed-out thinking block |

**Display behaviour:**

1. When `POST /api/chat` returns, immediately create an empty assistant message bubble with a typing cursor animation.
2. Open `EventSource` — each `chunk` appends text to that bubble.
3. On `done` — stop cursor animation, mark message complete.
4. On `error` — show error text in the bubble, style it differently (e.g. red border).

**Special rendering:** If the complete assembled text contains a fenced Mermaid block (` ```mermaid ... ``` `), render it as a diagram using a Mermaid library (e.g. `mermaid.js`). The backend generates valid Mermaid syntax for the `diagram` mode.

[
  {
    "thread_id": "abc12345",
    "title": "Binary Search Tree Explained",

### 6.4 `GET /api/sessions/{thread_id}/messages` — Load a conversation

Load the full message history to restore a past conversation.

```text
GET /api/sessions/abc12345/messages
```

**Response `200 OK`:**

```json
[
  { "role": "user", "content": "Explain binary search trees" },
  { "role": "assistant", "content": "A BST is a data structure where each node..." }
]
```

`role` is always either `"user"` or `"assistant"`. Render these with the same bubble components used for live chat.

---

### 6.5 `DELETE /api/sessions/{thread_id}` — Delete a conversation

```text
DELETE /api/sessions/abc12345
```

**Response `204 No Content`** — no body.

On success, remove the conversation from the sidebar list.

### 6.6 `POST /api/upload` — Upload documents

Upload one or more documents to the `user_uploads` collection.

```text
POST /api/upload
Content-Type: multipart/form-data
```

**Form fields:**

| Field | Type | Notes |
| --- | --- | --- |
| `files` | File[] | One or more files. Allowed: `.pdf`, `.docx`, `.pptx`, `.md`, `.txt` |
| `course` | string | Course to tag the upload to (or `"all"` for untagged) |

**Response `200 OK`:**

```json
{
  "status": "ok",
  "files_processed": 2,
  "chunks_indexed": 84
}
```

**Response `400 Bad Request`:**

```json
{ "detail": "File type .xlsx is not supported." }
```

**Response `413 Payload Too Large`:**

```json
{ "detail": "Upload limit is 100 documents." }
```

### 6.10 `GET /api/status` — System health

Poll this on app load (and every 5s while `model_ready` is false).

```text
GET /api/status
```

**Response `200 OK`:**

```json
{
  "model_ready": true,
  "model_name": "Qwen3.5-4B",
  "llm_port": 39821,
  "embedding_tier": "lite",
  "vectordb_collections": ["curriculum", "user_uploads"],
  "network_online": false
}
```

| Field | Meaning |
| --- | --- |
| `model_ready` | `false` during the ~5–10 second llama-server startup |
| `model_name` | Display in status bar / about page |
| `network_online` | `false` = fully offline mode (research/fix modes will be limited) |

**Loading state:** On first app open, show a full-screen loading screen until `model_ready: true`. Do not allow sending messages before this.

### 7.2 Screens / States

**Screen 1 — First Run (Name Setup)**  
Shown on very first launch only. A centered form with an emoji, heading "Welcome to Sage", subtext "Your offline AI study assistant", and a single name input + "Get Started →" button. Name is stored in `localStorage` (not sent to backend). Never shown again after submission.

**Screen 2 — Model Loading**  
Full-screen overlay while `/api/status` returns `model_ready: false`. Shows a spinner, the Sage logo, and "Getting ready..." text. Auto-dismisses when model is ready.

**Screen 3 — Welcome / Empty Chat**  
Shown when current thread has no messages. Centered layout, `"Hello, {name}! 👋"` as gradient heading, subtitle `"What would you like to study today?"`, and a row of mode chips (decorative, not functionally clickable — clicking selects that mode in the composer).

**Screen 4 — Active Chat**  
Standard chat interface: scrollable message list above, fixed composer bar below.

**Screen 5 — Settings / About** *(optional, can be a modal)*  
Shows institution name, model name, embedding tier, and app version.

### 7.3 Components

#### Sidebar

- Header: `🎓 Sage` logo + sidebar collapse toggle
- `✏️ New Chat` button — clears current thread, resets to Welcome screen
- Conversation list, grouped by recency (Today / Yesterday / This Week / Last Month)
  - Active conversation highlighted
  - Click → loads conversation via `GET /api/sessions/{thread_id}/messages`
  - Right-click or hover → Delete button (calls `DELETE /api/sessions/{thread_id}`)
- Divider
- `📁 My Docs` section — list of uploaded files, upload button (opens file picker), delete per file
- Divider
- Status indicator at bottom: green dot `● Model ready` or amber `○ Model loading...` + model name

#### Message Bubble — User

- Right-aligned
- Rounded pill shape (1.25rem radius, flat on bottom-right)
- Background: `#2b3445` (dark) / `#f3f4f6` (light)
- Max-width: 80% of content area

#### Message Bubble — Assistant

- Left-aligned, after a small avatar icon (`🎓` or a simple icon)
- Transparent background, just the text
- Full markdown rendering: bold, italic, code blocks (syntax highlighted), tables, lists
- Mermaid diagrams rendered inline (detect ` ```mermaid ` blocks)
- Max-width: 100% of content area

#### Composer Bar** (fixed bottom)

- Left: `✏️` popover button → opens mode selector panel (list of 8 modes, active one highlighted)
- Center: `textarea` — pill-shaped, auto-grows, placeholder changes with active mode (e.g. `💬 Ask Sage to explain...`)
- Right: `↑` send button — white circle on dark background
- Above the composer: active mode pill showing current mode (`💬 Explain`)
- Pressing Enter sends (Shift+Enter = newline)
- Disabled (greyed out) when `model_ready: false` or when a stream is in progress

#### Typing Cursor

- When streaming starts, show a blinking `▍` cursor at the end of the assembling text
- Remove it when `type: "done"` is received

#### Tool Call Indicator** *(optional but nice)*

- Small greyed-out line above the streaming bubble: `🔍 Searching course materials...`
- Driven by `type: "tool_call"` events. Map tool names:
  - `corpus_search` → `Searching course materials`
  - `calculator` → `Running calculation`
  - `web_search` → `Searching the web`
  - `arxiv_search` → `Searching arXiv`
  - `sandbox` → `Executing code`

## 8. Streaming Protocol (Detailed)

Call sequence for sending a message:

```text
1. User presses send
2. React: disable composer, show loading state
3. React: POST /api/chat  →  receives { thread_id, message_id }
4. React: append user bubble to message list
5. React: append empty assistant bubble with blinking cursor
6. React: open EventSource("/api/stream/{thread_id}")
7.   On data.type === "tool_call" → update status line inside bubble
8.   On data.type === "chunk"    → append data.text to bubble content
9.   On data.type === "thinking" → optionally show collapsible "Thought" block
10.  On data.type === "done"     → close EventSource, remove cursor, re-enable composer
11.  On data.type === "error"    → close EventSource, show error in bubble, re-enable composer
12. React: call GET /api/sessions to refresh sidebar (title may have updated)
```

**Important:** Only one stream can be active at a time. Disable the composer and mode selector during streaming. Do not call `POST /api/chat` again until the previous stream is complete.

## 9. Project Setup & Build Contract

**Tech stack (your choice, recommended):**

- Vite + React 18 + TypeScript
- No CSS framework required — use vanilla CSS or CSS Modules
- `react-markdown`, `rehype-highlight`, `remark-gfm`, `mermaid` for content rendering

**Directory location in the Sage workspace:**

```text
d:\Sage\frontend\        ← create this directory
  package.json
  vite.config.ts
  index.html
  src\
    ...
```

**Build output requirement:**

```bash
npm run build
```

Must produce `d:\Sage\frontend\dist\` with `index.html` at the root.

**Vite config (critical):**

```typescript
// vite.config.ts
export default {
  base: '/',
  build: {
    outDir: 'dist',
  },
  server: {
    proxy: {
      '/api': 'http://localhost:8765',  // dev-time only
    },
  },
}
```

During development, run `npm run dev` (Vite dev server on `:5173`) with the proxy above so API calls hit the Python backend. In production, FastAPI serves your compiled `dist/` directly — base URL becomes the same origin, no CORS needed.

**Do not hardcode `http://localhost:8501`** in fetch calls. Use relative paths (`/api/chat`) — they work in both dev (via proxy) and production (same-origin serving).

## 10. Mock Data for Development

Use this mock while the Python backend is not yet ready. Create `src/api/mock.ts`:

```typescript
export const MOCK_SESSIONS = [
  { thread_id: "aaa00001", title: "Binary Search Trees", last_message_preview: "A BST is a tree where...", updated_at: new Date().toISOString() },
  { thread_id: "aaa00002", title: "OOP Concepts in Python", last_message_preview: "Classes define...", updated_at: new Date(Date.now() - 86400000).toISOString() },
  { thread_id: "aaa00003", title: "Database Normalization", last_message_preview: "1NF requires...", updated_at: new Date(Date.now() - 86400000 * 3).toISOString() },
];

export const MOCK_STATUS = {
  model_ready: true,
  model_name: "Qwen3.5-4B",
  embedding_tier: "lite",
  network_online: false,
};

export const MOCK_COURSES = { courses: ["CS101", "CS201", "CS202", "CS301", "CS302"] };

// Mock stream: simulate token-by-token delivery
export function mockStream(onChunk: (text: string) => void, onDone: () => void) {
  const text = "Binary search is an efficient algorithm for finding an item in a **sorted** list.\n\n**Time Complexity:** $O(\\log n)$\n\nHere's an example:\n```python\ndef binary_search(arr, target):\n    lo, hi = 0, len(arr) - 1\n    while lo <= hi:\n        mid = lo + (hi - lo) // 2\n        if arr[mid] == target: return mid\n        elif arr[mid] < target: lo = mid + 1\n        else: hi = mid - 1\n    return -1\n```";
  const tokens = text.split(' ');
  let i = 0;
  const interval = setInterval(() => {
    if (i < tokens.length) { onChunk(tokens[i++] + ' '); }
    else { clearInterval(interval); onDone(); }
  }, 40);
}
```

Toggle between mock and real API with a single env var: `VITE_USE_MOCK=true`.
